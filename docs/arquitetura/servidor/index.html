Servidor

A arquitetura do lado servidor fora desenvolvida pensando em conver uma estrutura de fácil compreensão e que adote um padrão conhecido dentro do ambiente de desenvolvimento de aplicações web services.

Modelo MVC (Model-View-Controller)​

O design pattern adotado pela aplicação foi o modelo MVC (Model-View-Controller). MVC se trata de um modelo de fácil compreensão que define que o usuário que interage com a "View" (a interface da aplicação) executa as funções que são pertencentes ao lado do servidor através do uso de um "Controller", que é responsável por executar funções de validação, autenticação, entre outras. O "Controller" é também responsável interagir com o "Model" da aplicação, que, por sua vez, é responsável por interagir com o banco de dados através de seus métodos e devolver essas informações através de eventos e chamadas ao "View" da aplicação, fechando um ciclo por completo.

Note: podemos ter múltiplos Models, Views e Controllers dentro da aplicação para desenvolver as features necessárias para o esperado funcionamento da aplicação.

Estrutura de pastas​

Todo o código produzido de forma manual para o projeto está localizado dentro da pasta /src, outros arquivos localizados do lado de fora desta pasta como package.json são arquivos utilizados para definir configurações básicas do projeto, como dependências, variáveis de ambiente, entre outras.

Aqui estão os principais diretórios da aplicação localizados dentro de /src e que são de importante compreensão para manutenção e atualização do projeto:

  • server.ts: Esse arquivo é responsável por executar a função main da aplicação, que por sua vez, realiza conexões com as rotas, banco de dados, entre outros serviços, se tratando assim, do primeiro arquivo a ser executado quando o software é executado através de comandos como yarn start ou yarn dev.
  • /http/controllers: Dentro dessa pasta encontramos arquivos que dentro da arquitetura MVC são conhecidos como controladores, estes, por sua vez, são responsáveis por receber requisições do usuário através das rotas e retornar algum resultado.
  • /interfaces: Aqui dentro estão localizado alguns arquivos que são responsáveis por definir [interfaces](https://www.typescriptlang.org/docs/handbook/interfaces.html) que são utilizadas ao longo da aplicação.
  • /middlewares: Contém alguns arquivos com métodos que são responsáveis por conter métodos que são usados para realizar validações quando um controlador é chamado pela rota.
  • /models: Na arquitetura MVC, conhecida como modelo, essa camada é responsável por conter os modelos responsáveis por representar o domínio da aplicação, estes contêm um conjunto de métodos que interagem diretamente com a camada de dados.
  • /repositories: Dentro de repositories, nós encontramos métodos responsáveis por interagir com o modelo da nossa aplicação, que por sua vez carrega métodos que interagem diretamente com o banco de dados da aplicação.
  • /routes: Dentro de routes, temos arquivos que são basicamente responsáveis por fornecer um caminho para que o usuário interaja com os controladores.
  • /services: Dentro de services encontramos arquivos que são responsáveis por controlar e fornecer funções relacionadas a lógica de negócio da aplicação, se trata da camada que interage e fornece parâmetros aos repositórios.
  • /utils: Esse diretório contêm múltiplas funções que são úteis para prover funções genéricas para a aplicação, como formatadores de data ou de texto, possivelmente utilizadas ao longo de toda a aplicação.
